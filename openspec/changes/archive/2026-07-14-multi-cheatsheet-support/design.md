## Context

The app currently manages a single `Cheatsheet` object stored under a single localStorage key (`cheatsheet`). The preview renders shortcuts as individual cards in a grid, and editor inputs use single-line `<input>` fields. Users want to manage multiple independent cheatsheets, see a denser table-based preview, and enter multi-line action text.

## Goals / Non-Goals

**Goals:**
- Support creating, switching, renaming, and deleting multiple cheatsheets
- Persist each cheatsheet independently under its own localStorage key with a shared index
- Render preview sections as cards with inner `<table>` rows (keys + action columns)
- Support multi-line action text in AddShortcutForm and inline editing
- Sidebar (hamburger menu) for cheatsheet navigation

**Non-Goals:**
- No drag-and-drop reordering of cheatsheets
- No cloud sync or export/import beyond localStorage
- No keyboard shortcuts for sidebar navigation

## Decisions

**1. Storage: Index + per-cheatsheet keys**
- An index key `cheatsheet:index` stores `CheatsheetMeta[]` (id, title, updatedAt)
- Each cheatsheet stored as `cheatsheet:<id>` with the full `Cheatsheet` JSON
- Why: Allows loading just the index on startup (fast sidebar), individual cheatsheet load on switch, and independent save/delete operations without rewriting the entire collection

**2. Data model: `id` field on Cheatsheet + optional `CheatsheetMeta`**
- `Cheatsheet` gains `id: string` (generated by `crypto.randomUUID()`)
- `CheatsheetMeta` for the index: `{ id, title, updatedAt }`
- Why: IDs decouple identity from title/position; meta keeps the index lightweight

**3. Sidebar: Slide-in panel, not dropdown**
- A `<div>` that slides from the left on mobile/desktop, covers the toolbar
- Why: More scalable for many cheatsheets; follows the user's preference; familiar pattern

**4. Render: Replace ShortcutCard grid with section cards + tables**
- Keep `ShortcutCard` for inline editing in the preview, but restructure it as a `<tr>` element
- `LivePreview` renders each section as: `<div class="card">` → `<div class="card-header">` → `<table>` → `<ShortcutCardRow />`
- Why: Matches the referenced HTML design; table layout is denser and more readable for keyboard shortcut references

**5. Editor wrapping: `<textarea>` for action field only**
- Keys remain single-line `<input>` (shortcuts like `Ctrl+C` are inherently short)
- Action field becomes `<textarea rows={3}>` in both AddShortcutForm and ShortcutCard edit mode
- Display in preview uses `whitespace-pre-wrap` to render newlines

## Risks / Trade-offs

- **[Data migration]** Existing users have a single cheatsheet under the old key → migration function reads old key, creates a new cheatsheet with a generated ID, writes it to new format, removes old key. Run once on first load after update.
- **[Storage quota]** Multiple cheatsheets could hit localStorage quota (~5MB). Mitigation: reasonable limit; a single cheatsheet is tiny (JSON of a few KB).
- **[Sidebar complexity]** Sidebar state management (open/close, active selection) adds UI state. Mitigation: keep state local in App, pass down as props.
